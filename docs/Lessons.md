## 記録済みの教訓

### [2026-02-21] ファイル名の英語タイトル部分にアンダースコア不要

**状況:** 文書ファイルのリネーム時
**エラー内容:** `3626-QD-DHQGHN_Regulation_on_Undergraduate_Training_transcription.md` としてしまった
**原因:** タイトル部分もアンダースコア区切りにすると想定していた
**解決策:** タイトルはスペース区切りでそのまま: `3626-QD-DHQGHN_Regulation on Undergraduate Training_transcription.md`
**教訓:** 命名規則ではアンダースコアは Document ID とタイトルの「区切り」のみに使用。タイトル内はスペースのまま
**他プロジェクトへの応用可能性:** ❌

### [2026-02-21] 文書IDは番号だけでなく完全な形式を使用する

**状況:** DOC_REGISTRY のキーを `'3626'` としていた
**エラー内容:** ベトナムの文書IDは `{番号}-{種別}-{発行者}` が正式な形式。番号だけだと異なる発行者間で重複の可能性がある
**原因:** 文書IDの構造を正しく理解していなかった
**解決策:** `'3626-QD-DHQGHN'` のようにフルIDを使用。YAML では公式表記 `3626/QĐ-ĐHQGHN` を使い、ファイル名・JS キーではサニタイズ版を使用
**教訓:** ベトナム公文書のIDは `{番号}/{種別}-{発行者}` が正式。ファイルシステムでは `/` → `-`、ダイアクリティカル除去したサニタイズ版を使用
**他プロジェクトへの応用可能性:** ✅ ベトナムの公文書を扱うすべてのシステムで適用可能

### [2026-02-21] doc_id と doc_number の重複を避ける

**状況:** YAML Front Matter に `doc_id` と `doc_number` の2つのフィールドがあった
**エラー内容:** 両方とも文書番号を表しており冗長だった
**原因:** 設計時に用途の違いを明確にしていなかった
**解決策:** `doc_id` に統一。公式表記（`3626/QĐ-ĐHQGHN`）を値とし、`doc_number` は削除
**教訓:** メタデータフィールドは追加する前に既存フィールドとの重複を確認する
**他プロジェクトへの応用可能性:** ✅ メタデータスキーマ設計全般に適用可能

### [2026-02-21] Firebase Compat SDK を使用する（単一ファイル SPA の場合）

**状況:** Firebase SDK を `index.html` に追加する際
**エラー内容:** ES Module 版 Firebase SDK は `<script type="module">` が必要で、既存の IIFE スクリプトとスコープが分離される
**原因:** ES Module は独自スコープを持つため、`firebase` オブジェクトがグローバルに公開されない
**解決策:** Firebase Compat SDK（`firebase-app-compat.js` 等）を使用し、`firebase` をグローバルスコープに配置
**教訓:** 単一ファイル SPA（`<script>` タグ内に全 JS を記述）では、Compat SDK が最も統合しやすい
**他プロジェクトへの応用可能性:** ✅ ビルドステップなしの静的サイトで Firebase を使う場合全般

### [2026-02-21] replace_all でトレーリングスペースに注意

**状況:** 大量のテキスト置換を `replace_all` で実行した際
**エラー内容:** 末尾のスペースが意図せず削除された
**原因:** `old_string` にトレーリングスペースが含まれていた
**解決策:** 置換後にテキストを正規表現で検証し、意図しないスペース消失がないか確認
**教訓:** `replace_all` 使用時はトレーリングスペースに注意。大量置換後は差分を必ず確認
**他プロジェクトへの応用可能性:** ✅ テキスト処理・一括置換全般

### [2026-02-22] 文書IDはEN/JA版でも原文ベトナム語表記を維持する

**状況:** 翻訳版（EN/JA）で文書IDが翻訳・簡略化されていた
**エラー内容:** `4668/QĐ-ĐHQGHN` → `4668/QD-DHQGHN`（ダイアクリティカル除去）や `QĐ-VNU`（機関名翻訳）
**原因:** 翻訳AIがIDの一部を英語に変換してしまった
**解決策:** QAワークフロー Step 2.1.1 に「文書IDの原文表記維持」ルールを追加。検出パターンも整備
**教訓:** 公文書番号は固有名詞と同等。翻訳対象ではない。検出にはダイアクリティカル除去版と機関名翻訳版の両方をチェックする必要がある
**他プロジェクトへの応用可能性:** ✅ 多言語文書管理システム全般

### [2026-02-22] HTMLタグ内でMarkdown記法は使えない

**状況:** `<p align="center">*(Ban hành kèm theo...)*</p>` で `*...*` がレンダリングされずアスタリスクが表示された
**エラー内容:** HTMLブロック内ではMarkdown構文が処理されない
**原因:** Markdownレンダラーの仕様。HTMLタグ内はそのまま出力される
**解決策:** `*...*` → `<em>...</em>`、`**...**` → `<strong>...</strong>` に置換
**教訓:** HTMLタグ（`<p>`, `<div>` 等）内では必ずHTMLタグでフォーマットする。QA 1.4.2 で既にルール化済みだが見落としやすい
**他プロジェクトへの応用可能性:** ✅ Markdownファイル全般

### [2026-02-22] 送付先・署名欄はHTML flex divで2カラム配置が必須

**状況:** 3636の送付先（Nơi nhận）と署名欄がプレーンMarkdownで縦並びだった
**エラー内容:** PDFでは左右2カラム配置だが、MDでは再現されていなかった
**原因:** PDF Converter Appがプレーンテキストとして出力し、HTML化されていなかった
**解決策:** 3626のゴールドスタンダードに合わせて `display: flex; justify-content: space-between` で2カラム化
**教訓:** PDF→MD変換後、レイアウト要素（2カラム、署名欄、ヘッダー）は必ず手動でHTML化が必要。QA 1.4.10 に追加済み
**他プロジェクトへの応用可能性:** ❌

### [2026-02-22] khoản/điểm のインデントと境界認識

**状況:** PDFで `b)` の下にサブリストがあり、その後に `2.` が続く構造
**エラー内容:** `2. Phân công giảng dạy` が `b)` の子要素として誤認される可能性があった
**原因:** インデントが統一されていなかったため、階層構造が不明瞭
**解決策:** khoản=4sp、điểm=8sp、サブリスト=8sp+`- ` の統一ルール。リスト境界を正しく認識して、新しいkhoảnは必ずインデントレベルを戻す
**教訓:** PDFの視覚的レイアウトだけでなく、法令の階層構造（Điều→khoản→điểm→gạch đầu dòng）を正確に解析する必要がある。インデントは構造を反映するもの
**他プロジェクトへの応用可能性:** ✅ ベトナム法令文書の構造化全般

### [2026-02-22] 用語集に略語セクションを整備する

**状況:** 翻訳版で「KT. ディレクター 副社長」「ベトナム国家大学ハノイ校学長」等の誤訳が発生
**エラー内容:** `KT.` = `Ký thay`（代理署名）が「ディレクター」と訳された。`Giám đốc` = 総長が「学長」と混同された
**原因:** 略語と役職名が用語集に未登録だった
**解決策:** 用語集に略語セクション（11.1〜11.5）を追加。組織・文書種別・発行機関・署名欄・修士課程固有用語をカバー
**教訓:** 翻訳前に用語集の網羅性を確認する。特に略語・役職名・固有名詞は誤訳リスクが高い
**他プロジェクトへの応用可能性:** ✅ 多言語翻訳プロジェクト全般
